1. Основи Git

1. Git це система контролю версій, а GitHub це онлайн-платформа для роботи з Git.
2. Кожен користувач має повну копію репозиторію з усією історією.
3. git init створює новий локальний репозиторій, а git clone копіює існуючий з віддаленого сервера.

2. Staging Area та Commits

Що таке Staging Area і навіщо вона потрібна?
4. Зона, куди додаються зміни перед створенням коміту.
5. git add додає зміни в поточній папці, а git add -A у всьому проєкті.
6. Як по кайфу, або наприклад feat: add login form.
7. За допомогою git commit --amend.

3. Гілки та злиття

8. Гілка це вказівник на коміт, вона займає мало пам’яті й створюється миттєво.
9. merge об’єднує історії з усіма комітами, а rebase переносить зміни поверх іншої гілки.
10. Злиття без додаткового коміту, коли main просто пересувається вперед до останнього коміту гілки.
11. Коли однакові рядки змінено в різних гілках, уникнути можна роблячи дрібні коміти.
12. Тому що це змінює історію, а інші користувачі втратять синхронізацію зі своїми копіями.

4. Віддалені репозиторії

13. Стандартна назва віддаленого репозиторію, на який налаштовано з’єднання.
14. fetch завантажує оновлення, а pull одразу зливає їх у поточну гілку.
15. Перезаписує історію на сервері, що може знищити чужі коміти.
16. git push origin --delete branch-name.

5. Скасування змін

17. --soft залишає зміни в staging, --mixed повертає у робочу зону, а --hard повністю видаляє зміни.
18. reset змінює історію, а revert створює новий коміт, який відміняє попередній.
19. git restore --staged <file>.
20. git restore <file> або через git checkout.

6. Безпека

21. Паролі, токени, приватні ключі.
22. Плакать.
23. SSH зручніший і безпечніший, не треба кожний раз вводити пароль.
24. Безпечна альтернатива паролю для автентифікації в GitHub через HTTPS.

7. GitHub

25. Запит на злиття змін з однієї гілки в іншу після перевірки коду.
26. Issues для обговорення проблем чи ідей, а Pull Requests для пропозицій змін у коді.
27. Копія чужого репозиторію у свій акаунт для незалежної роботи над кодом.
28. Інструмент автоматизації.

8. Best Practices

29. Після кожної логічної зміни, щоб відслідковувати історію.
30. До реалізації функції, приблизно кілька днів.
31. Використовувати Git Flow або feature branch модель, окремі гілки для кожної задачі.
32. Ні, їх додають у .gitignore, бо їх можна встановити через npm install.
